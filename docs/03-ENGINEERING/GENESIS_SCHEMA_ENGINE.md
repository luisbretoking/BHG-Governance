# 1. Purpose

The Genesis Schema Engine validates that every Parsed Document Object conforms to the official structural schemas defined by the BHG Governance Framework.

The Schema Engine validates document structure only.

It SHALL never evaluate governance rules or business logic.

---

# 2. Mission

Guarantee structural consistency across the entire BHG documentation ecosystem.

---

# 3. Scope

The Schema Engine SHALL:

- validate document schemas;
- verify mandatory metadata;
- validate section hierarchy;
- validate required structural elements;
- report schema violations.

The Schema Engine SHALL NOT:

- interpret document meaning;
- validate governance policies;
- resolve dependencies;
- generate documentation;
- modify source documents.

---

# 4. Input

The Schema Engine SHALL receive:

- Parsed Document Object (PDO)

The original document SHALL never be processed directly.

---

# 5. Output

The Schema Engine SHALL produce a Schema Validation Report (SVR).

The report SHALL contain:

- schema identifier;
- validation status;
- detected violations;
- warnings;
- structural recommendations.

---

# 6. Schema Registry

Every schema SHALL have:

- schema identifier;
- schema version;
- document type;
- required metadata;
- mandatory sections;
- optional sections;
- validation rules.

---

# 7. Structural Validation

The engine SHALL validate:

- metadata completeness;
- heading hierarchy;
- section ordering;
- duplicate sections;
- mandatory sections;
- optional sections;
- code block placement;
- table structure;
- list consistency.

---

# 8. Metadata Validation

Mandatory metadata SHALL include:

- title;
- document_id;
- version;
- status;
- document_type;
- owner;
- language;
- classification;
- approved_by.

Missing mandatory fields SHALL produce validation failures.

---

# 9. Hierarchy Validation

Heading levels SHALL follow a valid hierarchical structure.

Skipped heading levels SHALL generate warnings.

Invalid heading nesting SHALL generate errors.

---

# 10. Schema Violations

Violations SHALL be classified as:

Information

Warning

Minor Violation

Major Violation

Critical Violation

Critical violations SHALL prevent downstream processing.

---

# 11. Schema Compatibility

Every schema SHALL be versioned.

Backward compatibility SHALL be explicitly declared.

Schema evolution SHALL preserve traceability.

---

# 12. Extensibility

New document schemas SHALL be registerable without modifying the Schema Engine.

Schemas SHALL be externalized from engine logic.

---

# 13. Performance Requirements

The Schema Engine SHALL be:

- deterministic;
- stateless;
- reproducible;
- implementation independent.

---

# 14. Security

The Schema Engine SHALL never:

- execute code;
- modify documents;
- bypass validation rules;
- download external resources.

---

# 15. Certification Requirements

A certified implementation SHALL demonstrate:

- deterministic validation;
- complete schema coverage;
- accurate metadata validation;
- hierarchy validation;
- reproducible reports.

---

# 16. Compliance

The Schema Engine SHALL comply with:

- BHG Constitution;
- Genesis Architecture;
- Genesis Runtime;
- Genesis Parser Engine;
- Engineering Constitution.

---

# 17. Final Provisions

The Genesis Schema Engine is the exclusive component responsible for structural validation.

Every downstream engine SHALL consume only Schema Validation Reports generated by the Schema Engine.
